## 一、机器清单（三台，别搞混）

| 角色 | hostname | 地址 | 系统 / 架构 | 连接方式 |
|------|----------|------|-------------|----------|
| **开发机**（SDK 机，跑 Claude / 交叉编译） | `ding-NUC12WSHi7` | `192.168.31.127`（wlo1 **WiFi**） | Ubuntu 22.04 x86_64 | 本机 |
| **哪吒 NUC**（采集 / 算法 / SPI 发送） | `ding-UP-ADLN01` | `192.168.31.79`（enp1s0 **有线**，DHCP 会变） | Linux 5.15 intel-iotg x86_64 | SSH `ding/1234` |
| **RK3568**（实时显示 / 电机 / 5G） | `ATK-DLRK3568` | 生产无网，靠串口 | Buildroot 内核 4.19 aarch64 | 串口 `/dev/ttyUSB0@1500000` |

> ⚠️ 开发机与哪吒**都在 192.168.31.x**，但是两台不同机器：
> - 开发机 = `.127`（WiFi wlo1），就是本机
> - 哪吒 = `.79`（有线 enp1s0，MAC `00:07:32:b6:49:dc`），DHCP 分配会变
> - 哪吒插**公网线**（如 101.7.144.x）时与开发机不同网段、**不可达**；必须插回 192.168.31.x 局域网。

## 二、快速连接

### 哪吒 SSH（paramiko，开发机无 sshpass）

```python
import paramiko
c = paramiko.SSHClient(); c.set_missing_host_key_policy(paramiko.AutoAddPolicy())
c.connect("192.168.31.79", username="ding", password="1234", timeout=8)
_, o, _ = c.exec_command("hostname; uname -srm")
print(o.read().decode()); c.close()
```
- 需要 root：`echo 1234 | sudo -S -p '' <cmd>`（**不要**在末尾加 `</dev/null`，会覆盖密码管道）。
- 后台进程：另开 session `exec_command` 发射后不读，或 `sudo -S sh -c 'nohup ... &'`。

### 找哪吒 IP（DHCP 变了时）

```bash
arp -an | grep -i "00:07:32:b6:49:dc"          # 按哪吒 MAC 找
# 或扫网段后用 hostname 确认是哪吒：
for i in $(seq 1 254); do ping -c1 -W1 192.168.31.$i >/dev/null 2>&1 && echo 192.168.31.$i; done
ssh ding@<ip> hostname    # 返回 ding-UP-ADLN01 即哪吒
```

### RK3568 串口

```bash
ls -l /dev/ttyUSB0          # 不在 = CH340 没插
picocom -b 1500000 /dev/ttyUSB0     # 人工登录, 退出 Ctrl-A Ctrl-Q
```
自动化用 pyserial（已装 3.5），CH340 须**慢写**（每 3 字节 sleep 6ms）。详见 `docs/rk3568_connection.md`。

## 三、SDK 交叉编译（开发机）

SDK 根：`~/rk3568_linux_sdk`（Buildroot SDK，与板 userspace 同源 Qt 5.15.2）。
**应用层一律用 Buildroot host 工具链 gcc 10.3.0，不用内核的 Linaro 6.3.1。**

```bash
SDK=~/rk3568_linux_sdk/buildroot/output/rockchip_rk3568
HOST=$SDK/host/bin           # qmake / aarch64-buildroot-linux-gnu-gcc(10.3.0) 都在这

# qt_display（Qt，qmake 自动接 sysroot）
cd ~/workspace/TOF3.0/rk3568/qt_display
$HOST/qmake qt_display.pro && make -j4
# 校验：file qt_display → ELF 64-bit ARM aarch64

# spi_receiver（C + 板上 /lib 的厂商 .so）
cd ~/workspace/TOF3.0/rk3568/spi_receiver
$HOST/aarch64-buildroot-linux-gnu-gcc -O2 -Wall -o spi_receiver spi_receiver.c \
    -Ideps -Ldeps/aarch64 -lUSB2UARTSPIIIC
```

> qmake 完整路径：`~/rk3568_linux_sdk/buildroot/output/rockchip_rk3568/host/bin/qmake`
> （注意中间是 `rockchip_rk3568`，旧文档写的 `output/host/bin` 路径不存在。）

产物经串口 base64 传板（CHUNK≈2000，慢写，md5 校验）。

## 五、实时显示链路 一键启动顺序

```
哪吒 sim_pf32 → /tmp/depth.dat → spi_syncer(root) → /dev/spidev1.0
  → USB-SPI适配器(0483:5740) → RK spi_receiver → /tmp/received.dat → qt_display → MIPI 屏
```

1. **哪吒**（SSH）：
   ```bash
   cd ~/tof3-rt/acquisition && nohup ./sim_pf32 >/tmp/sim_pf32.log 2>&1 </dev/null &
   echo 1234 | sudo -S -p '' sh -c 'nohup ~/tof3-rt/spi_syncer/spi_syncer >/tmp/spi_syncer.log 2>&1 </dev/null &'
   ```
2. **RK3568**（串口）：
   ```sh
   /etc/init.d/S95tof_spi_receiver restart   # 含 rmmod cdc_acm + USB unbind/bind
   /etc/init.d/S96tof_display restart         # 含 wait_for_weston
   ```
3. 验证：哪吒 `/tmp/depth.dat` 与 RK `/tmp/received.dat` 的 seq 都递增；qt_display 日志出现 `read frame ... seq=` 递增。

## 六、重要踩坑（务必遵守）

- **串口绝不发 Ctrl-C(`\x03`)**：板上 BusyBox sh 无 job control，`nohup &` 后台进程与 shell 同进程组，Ctrl-C 的 SIGINT 会连带杀掉 `spi_receiver`/`qt_display`。清行只用 Ctrl-U(`\x15`)。
- **CH340 慢写**：每 3 字节 `sleep 0.006`，否则丢字节。
- **哪吒 IP 是 DHCP**：会变，用 hostname / MAC 确认，别硬编码。
- **sudo 后台 + 日志**：`fs.protected_regular` 会让 root 无法在 /tmp 覆写**别的用户**拥有的文件 → 先 `rm` 旧 log 让 root 新建。
- **qt_display 黑屏自查**：日志有 `paintEvent hasDepthFrame=true` 但屏不动 → 多半是 **weston compositor 没在跑**（僵尸 wayland-0 socket），qt 在空转重绘。先确认 weston 进程，再重启 S96。
