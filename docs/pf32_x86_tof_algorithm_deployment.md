# PF32 x86 约束下的 ToF 成像与透雾算法部署建议

## 1. 核心判断

PF32 探测器依赖 x86 系统驱动/API，因此哪吒 x86 主机不只是采集节点，而应作为系统中的主要数据处理节点。RK3568 更适合作为控制、显示和轻量通信节点，不建议承担完整原始光子数据的大规模中转和上传压力。

推荐总体架构：

```text
PF32 探测器
  ↓
哪吒 x86：采集 + 预处理 + ToF/透雾算法 + 数据压缩
  ↓
RK3568：设备控制 + UI 显示 + 参数下发 + 轻量结果通信
  ↓
服务器/云端：实验数据存储 + 离线分析 + 可视化
```

不推荐架构：

```text
PF32 → 哪吒 x86 → 原始大数据 → RK3568 → 云端
```

原因是 RK3568 在该链路中会变成原始数据中转瓶颈，既增加传输压力，也没有充分利用哪吒 x86 已经靠近 PF32 驱动和原始数据源的优势。

## 2. 各节点功能分工

### 2.1 哪吒 x86

哪吒应作为采集与算法主节点，负责：

- 运行 PF32 官方驱动/API。
- 采集原始 photon histogram、raw frame 或 TCSPC 数据。
- 完成基础 ToF 成像流程：
  - 背景扣除
  - 阈值/CFAR 峰检测
  - 局部高斯拟合
  - 深度图、强度图、置信度图输出
- 完成复杂透雾算法：
  - Gamma/lognormal 雾散射峰建模
  - Gaussian 目标峰拟合
  - EM 或非线性最小二乘估计
  - DBSCAN 残差聚类
- 对输出结果进行压缩、筛选和分层上传。
- 本地保存完整实验原始数据，用于后续复现和算法调试。

### 2.2 RK3568

RK3568 不建议承担逐像素复杂拟合或完整原始数据上传。更适合负责：

- Qt UI 显示。
- 接收哪吒输出的深度图、强度图、置信度图和关键参数。
- 控制激光器、调焦电机、转台、串口外设等。
- 向哪吒或服务器下发采集参数、成像参数、调焦参数。
- 做轻量级显示增强和状态监控。

RK3568 可以承担的轻量算法：

- 简单背景扣除。
- 阈值峰检测。
- 简单深度换算。
- 低分辨率预览图显示。
- 参数状态判断。

不建议 RK3568 第一版承担：

- 逐像素 Gamma/Gaussian 非线性拟合。
- EM 迭代。
- DBSCAN 残差聚类。
- 大规模原始 histogram 上传中转。

### 2.3 服务器/云端

服务器或云端更适合负责：

- 实验数据归档。
- 离线算法对比。
- 参数扫描和模型调优。
- 多组雾浓度实验结果统计。
- Web 可视化和远程查看。

## 3. 数据上传分层策略

为避免 RK3568 或网络链路被原始数据压垮，建议采用三种数据模式。

### 3.1 实时模式

用于实时显示和系统运行。

上传内容：

```text
depth_map
intensity_map
confidence_map
fog_strength_map
system_status
key_parameters
```

特点：

- 数据量小。
- 适合实时显示。
- 适合 RK3568 和云端同步查看。

### 3.2 调试模式

用于算法调试和问题定位。

上传内容：

```text
ROI 区域完整 histogram
少量典型像素 histogram
异常像素 histogram
拟合参数
残差曲线
```

特点：

- 数据量中等。
- 保留算法诊断所需信息。
- 不需要每帧上传全量原始数据。

### 3.3 实验记录模式

用于论文实验、算法复现和离线分析。

保存内容：

```text
完整 raw histogram / raw.dat
采集时间戳
激光参数
PF32 参数
雾浓度参数
目标距离
算法输出结果
```

建议：

- 完整原始数据优先保存在哪吒本地硬盘。
- 实验结束后再批量上传服务器。
- 不走 RK3568 实时中转。

## 4. 第一版 ToF 成像算法

第一版建议先实现稳定基线：

```text
背景扣除 + CFAR/阈值峰检测 + 局部高斯拟合 + 深度/置信度输出
```

单像素 histogram 可以表示为：

```text
h(t) = background + target_echo(t)
```

处理流程：

```text
原始 histogram
  ↓
背景估计
  ↓
背景扣除
  ↓
阈值/CFAR 峰检测
  ↓
峰值局部高斯拟合
  ↓
ToF 深度换算
  ↓
depth_map + intensity_map + confidence_map
```

深度换算以 PF32 反向 start-stop 模式为例：

```text
ToF = (1023 - bin_index) × 55 ps
distance = ToF × c / 2
```

第一版目标不是追求最复杂模型，而是先跑通：

- 原始数据到深度图的完整链路。
- 有效像素判断。
- 置信度输出。
- 可视化和保存。

## 5. 透雾算法第一版

透雾场景下，histogram 更接近：

```text
h(t) = background + fog_backscatter(t) + target_echo(t)
```

第一版透雾模型建议：

```text
lognormal/gamma 雾散射峰 + Gaussian 目标峰 + EM/非线性拟合
```

推荐先使用 Gamma 雾散射模型：

```text
S_fog(t) = A_f · t^(k-1) · exp(-t/θ)
```

目标峰使用 Gaussian：

```text
S_target(t) = A_t · exp(-(t - μ_t)^2 / (2σ_t^2))
```

整体拟合：

```text
h(t) = B + S_fog(t) + S_target(t)
```

其中：

- `B`：背景噪声。
- `S_fog(t)`：雾后向散射。
- `S_target(t)`：目标回波。
- `μ_t`：目标峰位置，用于计算 ToF 深度。

该方法的意义是：强雾情况下，全局最大峰可能来自雾散射，而不是目标回波。通过显式建模雾峰，可以从雾散射中分离出目标峰。

## 6. 后续创新路线

后续可发展为：

```text
Gamma 背向散射模型 + DBSCAN 残差聚类 + Gaussian 目标峰拟合
```

核心思想：

```text
先用 Gamma 模型解释雾，
再从模型无法解释的残差中寻找目标回波。
```

流程：

```text
原始 histogram
  ↓
背景估计
  ↓
Gamma 雾散射拟合
  ↓
残差计算
  ↓
DBSCAN 残差峰聚类
  ↓
目标簇筛选
  ↓
Gaussian 目标峰拟合
  ↓
深度图 + 置信度图 + 雾强度图
```

残差定义：

```text
r(t) = h(t) - B - Gamma_fog(t)
```

如果存在目标，残差中会出现局部峰：

```text
r(t) ≈ Gaussian_target(t) + noise
```

DBSCAN 可用于从残差中自动寻找目标候选簇：

```text
X = {(t, r(t)) | r(t) > threshold}
```

目标簇筛选条件：

- 时间位置不能过早，避免误判雾峰。
- 峰宽不能过宽，避免误判散射尾。
- 峰值强度高于残差噪声。
- 峰面积达到最小光子数要求。
- 与邻域像素深度具有空间一致性。

最终对候选簇附近做 Gaussian 精拟合，得到更稳定的目标峰位置 `μ_t`。

## 7. 推荐落地顺序

建议按以下顺序开发：

1. 在哪吒 x86 上实现基础 ToF 成像链路：

```text
背景扣除 → 阈值/CFAR 峰检测 → 深度图/置信度图
```

2. 加入局部高斯拟合，提高峰值定位精度。

3. 实现 Gamma/lognormal 雾散射峰 + Gaussian 目标峰拟合。

4. 增加残差分析和 DBSCAN 聚类，用于强雾弱目标场景。

5. 将稳定后的轻量部分迁移到 RK3568：

```text
显示
控制
轻量深度图预览
参数下发
状态监控
```

6. 保留复杂模型在 x86 或服务器端运行。

## 8. 最终建议

PF32 驱动绑定 x86 后，系统设计应充分利用哪吒 x86 的位置优势和算力优势：

```text
哪吒 x86：采集 + 主算法
RK3568：控制 + 显示 + 参数交互
服务器：存储 + 离线分析
```

这样可以避免 RK3568 被原始数据中转压垮，同时也方便在 x86 上快速迭代透雾算法。
